# Milestone 2: Content Management — Design Specification

> **Created:** 2026-04-10
> **Status:** Draft
> **Depends on:** Milestone 1 (Auth & Database)

## Overview

Admin panel for managing collections and photos in the Kwelps Album gacha app. Users identified by `ADMIN_EMAIL` can create, edit, and delete collections and photos through a modern CMS-style interface.

## Goals

1. Allow admins to manage collections (CRUD)
2. Allow admins to manage photos with rarity assignment
3. Upload images directly to Supabase Storage
4. Control collection visibility via active/inactive toggle
5. Provide a clean, modern admin interface

## Non-Goals

- User-facing album view (Milestone 5)
- Pack purchase flow (Milestone 4)
- Wallet/balance management (Milestone 3)
- Complex role-based permissions (admin is determined by email only)

## User Flows

### Admin Authentication Flow

1. User logs in via standard login page
2. Middleware checks `user.email === process.env.ADMIN_EMAIL`
3. If admin → access to `/admin/*` routes
4. If not admin → redirect to `/dashboard` with "unauthorized" error

### Collection Management Flow

1. Admin navigates to `/admin/collections`
2. Sees list of all collections with status (active/inactive)
3. Can create new collection or edit existing
4. Can toggle `active` status with single click
5. Can delete collection (with confirmation if photos exist)

### Photo Management Flow

1. Admin navigates to `/admin/collections/[id]/photos`
2. Sees grid of photos in collection
3. Can upload new photos (single or batch)
4. Can assign rarity to each photo
5. Can delete photos (with confirmation)
6. Uploaded images go to Supabase Storage `photos` bucket

### Image Upload Flow

1. Admin selects file(s) via drag & drop or file picker
2. Client validates file type (jpg, png, webp) and size (max 5MB)
3. Client uploads directly to Supabase Storage
4. Supabase returns public URL
5. Server action creates Photo record with URL
6. Thumbnail URL generated by Supabase transformation

## API Design

### Server Actions

**Collections:**

| Action | Input | Output |
|--------|-------|--------|
| `createCollection` | `{ name, description? }` | `Collection` |
| `updateCollection` | `{ id, name?, description?, active? }` | `Collection` |
| `deleteCollection` | `{ id }` | `void` |
| `getCollections` | - | `Collection[]` |
| `getCollection` | `{ id }` | `Collection \| null` |

**Photos:**

| Action | Input | Output |
|--------|-------|--------|
| `createPhoto` | `{ collectionId, url, thumbnailUrl?, rarity }` | `Photo` |
| `updatePhoto` | `{ id, rarity?, thumbnailUrl? }` | `Photo` |
| `deletePhoto` | `{ id }` | `void` |
| `getPhotos` | `{ collectionId? }` | `Photo[]` |
| `uploadPhoto` | `FormData` (file + metadata) | `{ url, thumbnailUrl }` |

### Supabase Storage

**Bucket:** `photos`
- Public read access
- Write restricted to service role (admin actions)

**File naming:**
```
/photos/{collectionId}/original-{uuid}.{ext}
/photos/{collectionId}/thumb-{uuid}.{ext}
```

**Transformations:**
- Thumbnail: `?width=200&height=200&resize=cover`

## Security

### Row Level Security (RLS)

Add RLS policies for `collections` and `photos`:

```sql
-- Collections: readable by all authenticated, writable by admin only
ALTER TABLE collections ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Collections are readable by authenticated users"
  ON collections FOR SELECT
  USING (auth.uid() IS NOT NULL);

CREATE POLICY "Collections are writable by admin"
  ON collections FOR ALL
  USING (auth.jwt() ->> 'email' = current_setting('app.admin_email'));

-- Photos: readable by all authenticated, writable by admin only
ALTER TABLE photos ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Photos are readable by authenticated users"
  ON photos FOR SELECT
  USING (auth.uid() IS NOT NULL);

CREATE POLICY "Photos are writable by admin"
  ON photos FOR ALL
  USING (auth.jwt() ->> 'email' = current_setting('app.admin_email'));
```

### Input Validation

All server actions validate input with Zod schemas:

```typescript
// collection.ts
export const collectionSchema = z.object({
  name: z.string().min(1).max(100),
  description: z.string().max(500).optional(),
  active: z.boolean().optional(),
});

// photo.ts
export const photoSchema = z.object({
  collectionId: z.string().uuid(),
  url: z.string().url(),
  thumbnailUrl: z.string().url().optional(),
  rarity: z.enum(['COMMON', 'RARE', 'EPIC', 'LEGENDARY']),
});
```

## UI Components

### Admin Layout

- Sidebar navigation (Dashboard, Collections, Photos)
- Header with admin badge
- Mobile responsive (sidebar collapses on small screens)

### Collection List

- Table with columns: Name, Description, Status, Photo Count, Actions
- Status toggle (active/inactive) as switch
- Quick actions: Edit, Manage Photos, Delete
- Empty state with "Create Collection" CTA

### Collection Form

- Name input (required, max 100 chars)
- Description textarea (optional, max 500 chars)
- Active toggle
- Save/Cancel buttons

### Photo Grid

- Responsive grid (4 columns on desktop, 2 on mobile)
- Each card shows: thumbnail, rarity badge, delete button
- Drag & drop upload zone at top
- Rarity dropdown for each photo
- Batch select for bulk actions

### Photo Uploader

- Drag & drop zone with file type indicators
- Progress indicator during upload
- Preview before confirming
- Error states (file too large, wrong type)

## Error Handling

| Scenario | Handling |
|----------|----------|
| Unauthorized access | Redirect to dashboard with toast |
| Invalid input | Form validation errors inline |
| Upload failure | Toast with retry option |
| Network error | Toast with generic error message |
| Delete with children | Confirm dialog, option to cascade or cancel |

## Implementation Notes

1. **Supabase Storage Setup:** Create `photos` bucket with public read policy
2. **Admin Middleware:** Extend existing middleware to check `ADMIN_EMAIL`
3. **Image Optimization:** Use Next.js Image component with Supabase transformation URLs
4. **Client Components:** Only photo-uploader and photo-grid need `'use client'`
5. **Server Actions:** All CRUD operations use server actions for security
